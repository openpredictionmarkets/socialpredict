# Embeddable Pages And Market Sharing Design

## Purpose

This document translates [03-embeddable-pages-and-market-sharing.md](./03-embeddable-pages-and-market-sharing.md) into a system design for iframe embedding and Open Graph market sharing.

It is not the implementation plan. The implementation sequence lives in [PLAN.md](./PLAN.md).

## Design Inputs

Primary inputs:

- [03-embeddable-pages-and-market-sharing.md](./03-embeddable-pages-and-market-sharing.md)
- Canonical design plan: `spec-socialpredict-tasks-auto/lib/design/design-plan.json`
- Designer-agent postures from `spec-socialpredict-tasks-auto/.codex/agents/`
- OpenGraph.io Open Graph metadata reference: <https://www.opengraph.io/open-graph-meta-tags>
- Existing frontend SPA and backend market APIs
- Existing deployment security-header and reverse-proxy setup

The canonical design plan remains the repository-level design source of truth. This feature design must conform to it rather than create a competing architecture.

## Designer Lens Review

Evans/domain lens:

- Embedding is an operator/publication concern, not just a frontend toggle.
- Market sharing must use market-domain language: title, description, public status, canonical URL, and public visibility.
- `public market`, `private proposal`, `embed host`, `share card`, and `canonical market URL` need stable meanings.

Fowler/evolutionary lens:

- Start with policy-controlled iframe support and market detail share metadata before designing a full widget platform.
- Decide metadata delivery based on crawler behavior, not aesthetics.
- Keep implementation reversible: headers and share shell routes can evolve without rewriting market domain policy.

Martin/clean-architecture lens:

- Frontend route code must not become the source of truth for market visibility or preview content.
- Backend or edge adapters may render metadata, but they should consume public market data through explicit seams.
- Security headers are deployment/runtime details, but they encode operator policy and must not be silently bypassed by page components.

## Problem Framing

SocialPredict URLs should travel well across the web. Today, a market link can load in the app, but it is not guaranteed to produce a useful social preview. Separately, partner sites may want to embed SocialPredict directly, but browser frame protections and clickjacking concerns make this a deployment and security-policy decision.

The feature must improve distribution while preserving public/private market boundaries and operator control over embedding.

## Business Outcomes

This feature should enable:

- Better market virality through social and messaging previews.
- Easier distribution through iframe embedding on trusted or public host pages.
- Fewer support questions when market links are shared externally.
- A foundation for future widgets without prematurely building a widget SDK.
- Safer environment-specific embed posture through explicit headers and documentation.

## Out Of Scope

This design does not initially introduce:

- A widget SDK.
- White-label partner theming.
- Signed embeds.
- Personalized previews.
- SSR for the entire application unless market metadata delivery requires it.
- Embeddable admin dashboards without additional design.

| Embedded page | A SocialPredict page displayed inside another site's iframe. | Native SocialPredict route. |
| Embed host | External page/domain that frames SocialPredict. | SocialPredict host. |
| Frame ancestor | Browser/CSP term for an allowed parent document that can embed the app. | API caller. |

| Share card | Link preview generated by social platforms or messaging clients from metadata. | In-app market card. |

| Share metadata | Public read model containing only market fields approved for link-preview rendering. | Full market record or admin queue row. |

## Context Map

Iframe embedding path:

- Operator config determines allowed frame ancestors.
- Runtime/proxy emits frame-related headers and is the source of truth for `Content-Security-Policy: frame-ancestors`.
- Browser enforces whether the embed host may frame SocialPredict.
- Frontend presents the route inside the iframe if allowed.
- Backend remains authoritative for authenticated actions and market state.

Market sharing path:

- Crawler requests a market URL.
- Server, edge, or share shell emits Open Graph metadata from public market data.
- Social platform builds a share card.
- Human follows the canonical URL into the SPA.

## Core Design Rules

Embedding rules:

- Embedding must be controlled by operator/environment policy.
- Frame-related headers must be explicit and testable.
- Removing `X-Frame-Options` is not enough; CSP `frame-ancestors` must be considered.
- Runtime/proxy configuration owns final frame-header composition; backend share shells and frontend pages must not override it independently.
- The first implementation should be default-deny and route-restricted to public, read-only pages unless a deliberate environment policy says otherwise.
- Admin, account-changing, and trading surfaces need explicit clickjacking review before broad embedding.
- If site-wide embedding is enabled, the risk acceptance should be documented.

Market-sharing rules:

- Public market pages should emit `og:title`, `og:type`, `og:url`, `og:description`, and `og:image`.
- `og:url` should be the canonical public market URL, not an API URL.
- `og:title` should use the market title, length-bounded for share clients.
- `og:description` should use a sanitized public summary or description.
- `og:image` should be an absolute URL to a stable image.
- Proposed/rejected moderator markets should not emit public market share metadata unless their visibility policy changes.
- Cancelled or private markets should not emit market-specific public share metadata.
- Resolved or closed markets may emit metadata only if they remain public market detail pages.

Metadata delivery rules:

- Do not assume client-side React meta updates are enough for crawlers.
- Prefer a backend/edge/share-shell strategy that emits metadata in initial HTML.
- Use an explicit share metadata read model or helper, not admin or moderator queues, as the metadata source.
- Missing or non-public market pages should return safe fallback metadata or safe not-found behavior.
- Canonical URLs and image URLs should come from typed public base URL configuration so environments do not hard-code production URLs.
- Document crawler cache/staleness expectations because social preview services may continue showing older metadata after market edits.

## Share Metadata Seam

The preferred internal seam is a small public read model, for example:

```text
ShareMetadata {
  market_id
  title
  description
  canonical_url
  image_url
  public_status
}
```

Rules for the seam:

- Prediction Market owns whether a market is public/shareable.
- API/Auth owns exclusion of private, admin, moderator-queue, and account data.
- Runtime/Configuration owns public base URL and image URL configuration.
- Backend share-shell or proxy rendering consumes `ShareMetadata`; it does not inspect full market records directly.
- Non-public or missing markets return a safe not-found/fallback result through the normal failure translation boundary.

## Candidate Implementation Options

### Option A: Backend Share Shell

Backend serves HTML for market routes with Open Graph tags and then loads the SPA.

Pros:

- Strong crawler compatibility.
- Clear backend access to public market data.
- No full frontend SSR required.

Cons:

- Requires route/proxy coordination for `/markets/{id}`.
- Backend emits some frontend-adjacent HTML.

### Option B: Reverse Proxy Metadata Injection

Proxy or edge layer fetches public market data and injects metadata.

Pros:

- Keeps SPA bundle mostly unchanged.
- Can be isolated at deploy edge.

Cons:

- More operational complexity.
- Easy to create duplicated data-fetching semantics outside the app.

### Option C: Client-Side Meta Updates Only

React updates document metadata after route load.

Pros:

- Simple frontend implementation.

Cons:

- Many crawlers will not execute it reliably.
- Not sufficient as the primary design unless proven by target preview tests.

Recommended first direction: Option A, unless current deployment routing makes it materially harder than an edge shell. Option A should consume the share metadata seam above rather than embedding market visibility rules in HTML rendering code.

## Security And Testing Strategy

Security tests should verify:

- Expected frame headers are present per environment.
- Disallowed origins cannot embed the app when policy is restricted.
- Admin/private pages do not accidentally emit permissive frame policy if excluded.
- Authenticated account-changing and trading routes remain denied unless explicitly approved.
- Non-public markets do not produce public share metadata.

Sharing tests should verify:

- Generated market HTML includes required Open Graph tags.
- Tag values are escaped and length-bounded.
- URLs and images are absolute.
- A sample market URL works in an external Open Graph preview tester.

## Open Questions

- Which specific frame ancestors should be allowed per environment?
- Should demo/staging have a more permissive embed policy than production?
- Should market share images be static defaults, generated images, or market-specific images?
- Should share metadata include market status, probability, volume, or only title/description?
- Should the first implementation use a backend share shell or proxy-level metadata injection?
